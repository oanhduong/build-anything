<objective>
Build the RETR-01 gate: a fresh-benchmark harness that measures whether grep-based
retrieval is a measurable bottleneck (miss-rate or latency), and writes
`gate-evidence.md` ONLY when a threshold is crossed. This is the hard gate — no
Phase 4 vector-index work (Plans 02-03) may proceed unless this script exits 0 and
produces the evidence file.

Purpose: The "evidence-first" core constraint (PROJECT.md) forbids building heavy
retrieval without proof. The critical research finding (RESEARCH.md Pitfall 1) is
that trace.log does NOT record search misses (a 0-match grep pipeline still exits 0),
so the gate CANNOT be derived from history — it MUST run a fresh benchmark against
the live grep command.

Output: `scripts/check-retrieval-gate.sh`, `scripts/retrieval/benchmark.py`, and
(conditionally, only if the gate opens) `gate-evidence.md`.

    Create `scripts/retrieval/benchmark.py` — a standalone Python script the gate
    shell calls. It runs a FRESH grep benchmark (it does NOT read ~/.claude/trace.log).

    CLI contract (called by check-retrieval-gate.sh):
      `python3 scripts/retrieval/benchmark.py --corpus-dir <DIR> --queries-file <FILE>`

    Behavior:
    1. Read newline-delimited queries from `--queries-file` (one query per line).
    2. For each query, run the CURRENT retrieval command against `--corpus-dir`:
       `grep -rn "<query>" <corpus-dir> 2>/dev/null` (use subprocess.run with a list,
       capture stdout; DO NOT pipe to head — we need the true result count).
       - Count result lines (len of non-empty stdout lines). 0 lines == a MISS.
       - Time each grep call with `time.perf_counter()` (start before subprocess.run,
         stop after). Record milliseconds = (end - start) * 1000.
    3. Compute:
       - `miss_rate` = (# queries with 0 results) / (total queries), as a fraction 0.0-1.0
       - `avg_latency_ms` = mean of per-query milliseconds
       - `corpus_size` = count of *.md files under `--corpus-dir` (recursive)
    4. Print results as machine-parseable KEY=VALUE lines to stdout, EXACTLY these keys:
       ```
       CORPUS_SIZE=<int>
       QUERY_COUNT=<int>
       MISS_RATE=<float with 4 decimals>
       AVG_LATENCY_MS=<float with 2 decimals>
       ```
    5. Exit 0 on success; exit 1 on any error (missing dir/file) with a message to stderr.

    Do NOT import chromadb or any vector library here — this benchmark measures the
    GREP baseline only. No embedding model is loaded.

    Keep it pure-stdlib (subprocess, time, pathlib, argparse, statistics) — no pip deps,
    so the gate can run before any chromadb install.
  </action>
